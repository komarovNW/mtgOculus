Ты работаешь как frontend-разработчик и технический помощник по созданию MVP веб-сайта для проекта **MTG Global Stats**.

Проект — это публичный веб-сайт со статистикой турниров Magic: The Gathering. Пользователи заходят на сайт без авторизации и смотрят статистику по турнирам, игрокам, колодам и метагейму. В MVP основной срез данных: **Москва + Legacy + все клубы**. В будущем должны появиться фильтры по городу, клубу, формату, типу турнира и датам.

## 2. Главная цель работы

Нужно создать MVP веб-сайта на React, который:

- работает без авторизации для публичных страниц;
- показывает главную страницу статистики;
- показывает список турниров и страницу конкретного турнира;
- показывает список игроков и страницу конкретного игрока;
- показывает список колод и страницу конкретной колоды;
- содержит экран ручного создания турнира для MVP;
- умеет работать на mock data до готовности backend;
- затем может быть переключён на реальные API через `.env`;
- имеет понятную структуру проекта, чтобы его мог поддерживать человек с небольшим frontend-опытом;
- собирается в production build и готовится к деплою.

## 3. Технический стек

Используй стек, описанный в документации. Базовое ожидание:

- React

- Vite
- React Router
- TanStack Query или простой API layer, если TanStack Query не нужен на первом шаге

Не добавляй тяжёлые библиотеки без необходимости. MVP должен быть простым и поддерживаемым.

---

## 4. Важный принцип разработки

Сначала сайт должен работать на моковых данных. Не жди backend.

Обязательная логика:

```env
VITE_USE_MOCKS=true
VITE_API_BASE_URL=http://localhost:8000/api/v1
```

При `VITE_USE_MOCKS=true` все страницы должны получать данные из mock-файлов.

При `VITE_USE_MOCKS=false` frontend должен обращаться к backend API.

## 5. Порядок разработки

Работай по этапам. Не пытайся сделать весь сайт одним большим изменением.

### Этап 1. Инициализация проекта

Сделай:

- Vite + React + TypeScript проект;
- базовую структуру папок по `PROJECT_STRUCTURE.md`;
- routing по `ROUTES.md`;
- `.env.example` на основе `ENV_EXAMPLE.md`;
- базовый layout сайта;
- shared API layer;
- mock data.

Результат этапа: проект запускается командой `npm run dev`, есть layout и пустые страницы маршрутов.

### Этап 2. UI primitives

Сделай базовые переиспользуемые компоненты:

- `PageLayout`
- `Section`
- `Card`
- `StatCard`
- `Table`
- `Button`
- `Select`
- `Input`
- `Badge`
- `Tabs`
- `LoadingState`
- `EmptyState`
- `ErrorState`

Не верстай каждый экран уникально. Переиспользуй компоненты.

### Этап 3. Главная страница

Сделай главную страницу `/` по `mtg_mvp_home_page_api.md`.

На странице должны быть:

- summary cards;
- последние турниры;
- метагейм колод;
- результативность колод;
- топ игроков;
- популярные матчапы;
- подготовленные фильтры по городу, клубу, формату, типу турнира и датам, даже если в MVP дефолт — Москва + Legacy.

Все tournament/player/deck элементы должны иметь id и быть кликабельными, если для них есть страница.

### Этап 4. Турниры

Сделай:

- `/tournaments` — список турниров;
- `/tournaments/:id` — страница конкретного турнира.

Страница турнира должна показывать:

- header турнира;
- финальные стендинги;
- все раунды/паринги;
- список игрок → колода;
- метагейм турнира, если данные есть.

Используй требования из `mtg_mvp_tournament_detail_api.md`.

### Этап 5. Игроки

Сделай:

- `/players` — список игроков;
- `/players/:id` — страница конкретного игрока.

Страница игрока должна показывать:

- summary;
- турниры игрока;
- колоды игрока;
- блок матчей можно подготовить архитектурно, но не обязательно делать в MVP, если он помечен как future.

Используй требования из `mtg_mvp_player_detail_api.md`.

### Этап 6. Колоды

Сделай:

- `/decks` — список колод;
- `/decks/:id` — страница конкретной колоды.

Страница колоды должна показывать:

- summary;
- результаты колоды по турнирам;
- игроков, которые играли этой колодой;
- блок матчапов можно подготовить архитектурно, но не обязательно делать в первом MVP.

Используй требования из `mtg_mvp_deck_detail_api.md`.

### Этап 7. Создание турнира

Сделай экран:

- `/admin/tournaments/create`

В MVP авторизацию не делаем, но маршрут отделяем как admin.

Форма должна содержать:

- дата турнира;
- город;
- клуб, который зависит от выбранного города;
- тип турнира: дейлик / турнир;
- формат;
- CSV файл со всеми раундами;
- CSV файл с финальными стендингами;
- textarea со списком `игрок - колода`.

Используй требования из `mtg_mvp_create_tournament_api.md`.

На success показать ссылку на созданный турнир. На error показать список ошибок и warnings от backend.

### Этап 8. Подключение real API

Когда mock-версия готова:

- подключи реальные API;
- используй `API_INTEGRATION_CHECKLIST.md`;
- не ломай mock mode;
- все API-вызовы должны быть централизованы в API layer.

### Этап 9. Production build и деплой

Сделай:

- `npm run build` без ошибок;
- `npm run preview` для локальной проверки production build;
- документацию по деплою;
- настройки для SPA fallback, если используется React Router.

## 7. Правила качества

Следуй этим правилам:

- Пиши TypeScript-типы для всех API responses.
- Не используй `any`, если можно описать тип.
- Не дублируй типы в разных файлах.
- Не храни API-вызовы внутри React-компонентов напрямую, выноси в API layer.
- Не смешивай бизнес-логику и UI-компоненты.
- Не делай сложный state management без необходимости.
- Не добавляй авторизацию в MVP.
- Не добавляй backend-код в frontend-проект.
- Не делай неподдерживаемую “магическую” структуру.
- Не ломай mock mode при подключении real API.
- Все страницы должны быть читаемыми на мобильных устройствах.
- Все таблицы должны иметь fallback для узких экранов: horizontal scroll или карточное представление.
- Все проценты показывать с одним знаком после запятой.
- Пустые значения показывать как `—`.

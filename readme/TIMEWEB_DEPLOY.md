# Деплой в Timeweb App Platform

Актуально для проекта `Magic Oculus`.

Репозиторий:

```text
git@github.com:komarovNW/mtgOculus.git
```

## Рекомендуемый сценарий

Для текущего проекта используем:

- `App Platform`
- тип приложения: `Frontend`
- фреймворк: `React`
- ветка: `main`
- Node.js: `22`
- build command: `npm run build`
- output directory: `dist`

## Как сделать автодеплой после push

Чтобы Timeweb обновлял релиз после каждого push в `main`, должны выполняться оба условия:

1. Репозиторий подключён через интеграцию аккаунта `GitHub`, а не просто через URL.
2. В настройках приложения включена опция:

```text
Сборка по последнему выполненному коммиту
```

Если оба условия выполнены, flow такой:

1. Разработчик пушит коммит в `origin/main`.
2. Timeweb видит новый коммит в подключённой ветке.
3. Стартует новый build/deploy.
4. После успешной сборки новая версия становится production-релизом.

Важно: версия обновляется не в момент push, а после успешного завершения сборки на стороне Timeweb.

## Когда автодеплой не сработает

Автоматического обновления не будет, если:

- репозиторий подключён по `URL`, а не через GitHub integration;
- выключен переключатель `Сборка по последнему выполненному коммиту`;
- push ушёл не в ту ветку;
- build упал на этапе установки зависимостей или сборки;
- в Timeweb сохранена другая ветка, а изменения пушатся в `main`.

## Если нужен ручной релиз

Если вы хотите выкатывать изменения не после каждого push, а вручную:

1. Отключите `Сборка по последнему выполненному коммиту`.
2. Продолжайте пушить изменения в GitHub.
3. Запускайте deploy вручную из вкладки `Деплой`, когда коммит готов к релизу.

Такой режим полезен, если:

- `main` ещё не является release-веткой;
- в репозиторий часто пушатся промежуточные изменения;
- перед выкладкой нужен ручной smoke-check.

## Рекомендуемый рабочий процесс

Для этого проекта безопаснее всего так:

1. Работать в feature-ветках.
2. Проверять изменения локально.
3. Мержить в `main` только готовые коммиты.
4. Держать в Timeweb автодеплой включённым.

Тогда:

- `main` становится релизной веткой;
- каждый push в `main` автоматически обновляет production;
- история деплоев в Timeweb совпадает с историей релизных коммитов.

## Переменные окружения в Timeweb

### Временный запуск на моках

```env
VITE_API_BASE_URL=http://localhost:8000/api/v1
VITE_USE_MOCKS=true
VITE_MOCK_SCENARIO=default
VITE_ADMIN_LOGIN=admin
VITE_ADMIN_PASSWORD=123456
VITE_ADMIN_DISPLAY_NAME=Администратор
```

### Когда появится backend

```env
VITE_API_BASE_URL=https://your-api-domain/api/v1
VITE_USE_MOCKS=false
```

Если backend-auth ещё не подключён, но нужен доступ к служебному разделу на стенде, временные `VITE_ADMIN_*` переменные тоже должны быть заданы.

## Что проверить после деплоя

- открывается `/`;
- работают публичные страницы;
- работает SPA-routing после прямого открытия detail page;
- если включён mock-auth, открывается `/login` и пропускает на `/admin/tournaments/create`;
- тема `Светлая / Тёмная` переключается без визуальных поломок.

## Откат

Если новый релиз оказался неудачным:

1. Откройте вкладку `Деплой`.
2. Найдите предыдущий успешный коммит.
3. Запустите redeploy этого коммита.

Для этого полезно держать короткие осмысленные commit messages.

# UI Guidelines

Текущий интерфейс проекта должен выглядеть как спокойный аналитический dashboard, где главный фокус на таблицах, фильтрах, карточках и понятных текстах, а не на декоративных эффектах.

## 1. Базовый принцип

Сайт делается для игроков и организаторов MTG, но визуально он должен оставаться ближе к статистическому веб-сервису, чем к игровому лаунчеру.

Что важно:

- данные читаются быстро;
- таблицы остаются главным форматом просмотра;
- текст помогает понять смысл метрик;
- цвета не спорят с контентом.

## 2. Темы

В проекте есть две темы:

- `light` — основная по умолчанию;
- `dark` — дополнительная спокойная тёмная тема.

Переключатель темы находится в шапке и сохраняет выбор пользователя локально.

## 3. Дизайн-токены

Все основные цвета должны идти через CSS variables в `src/app/styles/globals.css`.

Базовые токены:

- `--color-bg`
- `--color-surface`
- `--color-surface-muted`
- `--color-surface-hover`
- `--color-border`
- `--color-border-strong`
- `--color-text-primary`
- `--color-text-secondary`
- `--color-text-muted`
- `--color-accent`
- `--color-accent-soft`
- `--color-success`
- `--color-warning`
- `--color-danger`
- `--color-chart-1 ... --color-chart-6`

Компоненты не должны хардкодить случайные hex-цвета напрямую.

## 4. Светлая тема

Светлая тема — дефолтная и основная для production.

Ожидание:

- фон страницы почти белый или очень светло-серый;
- карточки и таблицы белые;
- границы тонкие и нейтральные;
- основной текст тёмный;
- вторичный текст серый;
- акцент один и спокойный: синий.

## 5. Тёмная тема

Тёмная тема нужна для долгого чтения таблиц, а не для “вау-эффекта”.

Ожидание:

- фон тёмный, но не чёрный;
- карточки чуть светлее фона;
- разделители читаемые;
- текст светлый, но не кислотный;
- акцент тот же синий, адаптированный под dark theme.

## 6. Таблицы

Таблицы — главный формат отображения данных.

Требования:

- заголовок таблицы отделён спокойным фоном;
- строки имеют тонкие разделители;
- hover у строк мягкий и не отвлекает;
- числовые значения читаются без лишних эффектов;
- сортировка по колонкам выглядит аккуратно и предсказуемо;
- secondary text не конкурирует с primary text;
- данные не должны “плавать” из-за слишком больших паддингов или декоративных блоков.

Если таблица кликабельна или содержит ссылки:

- hover должен подсказывать интерактивность;
- ссылки на турнир, игрока и колоду должны визуально читаться как ссылки, но без агрессивного оформления.

## 7. Карточки и summary blocks

Карточки должны быть нейтральными контейнерами под данные.

Правила:

- не красить каждую карточку в свой цвет;
- использовать минимальные тени или обходиться без них;
- опираться на `surface`, `border`, `text`, а не на градиенты;
- заголовок карточки короткий и понятный;
- сложная метрика получает маленькую подсказку, а не длинный технический текст.

## 8. Графики

Если используется график:

- основная палитра берётся только из `--color-chart-*`;
- график не заменяет таблицу полностью;
- рядом всегда должны быть текст и числа;
- для метагейма при большом количестве колод предпочтительнее bar chart, а не pie chart.

## 9. MTG-цвета

MTG-цвета допустимы только как маленькие акценты:

- mana pips;
- chips `W/U/B/R/G/C` рядом с колодой.

Их нельзя использовать как основу всего интерфейса.

## 10. Тексты интерфейса

Tone of voice:

- коротко;
- понятно;
- по-человечески;
- на русском;
- с MTG-терминами там, где они действительно помогают.

Не использовать:

- технические названия API-полей;
- машинный перевод;
- канцелярит;
- сухие админские заглушки.

Хороший пример:

- `Пока нет турниров по выбранным фильтрам`
- `Не получилось загрузить статистику`
- `Результат матчей`
- `Любимая колода`

## 11. Состояния

### Loading

Используем человеческие тексты:

- `Загружаем статистику...`
- `Собираем данные турниров...`

### Empty

Состояние без данных должно объяснять, что происходит:

- `Пока нет данных по этому набору фильтров`
- `Турниры по этим фильтрам ещё не загружены`

### Error

Ошибка должна быть понятной:

- `Не удалось загрузить данные`
- `Попробуйте обновить страницу или изменить фильтры`

## 12. Tooltip и info hints

Подсказки нужны только там, где метрика неочевидна.

Подсказки уже уместны для:

- процента побед;
- малой выборки;
- количества участий;
- результата матчей, если формат нужно пояснить.

Tooltip не должен заменять нормальный заголовок. Сначала понятный текст, потом короткое уточнение.

## 13. Адаптивность

Интерфейс должен уверенно работать на:

- desktop;
- tablet;
- mobile.

Что важно:

- таблицы не ломают layout;
- на узких экранах допустим горизонтальный scroll;
- отступы по краям экрана должны быть экономными и давать максимум места таблице;
- длинные названия не должны разрушать сетку.

## 14. Чего не делать

- не использовать неон, киберпанк и “игровой” UI;
- не строить интерфейс на градиентах;
- не раскрашивать каждую карточку отдельно;
- не превращать страницу в набор декоративных блоков;
- не прятать важные метрики за длинными пояснениями;
- не копировать raw API naming в UI.

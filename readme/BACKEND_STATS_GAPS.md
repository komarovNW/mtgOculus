# Чего не хватает backend для текущего показа статистики

Этот документ не заменяет `BACKEND_API_HANDOFF.md`. Он нужен как короткий рабочий список:
что ещё должно появиться в backend, чтобы frontend показывал статистику ровно в том виде,
в котором она уже сверстана сейчас.

Ниже перечислены только пробелы. Всё, чего здесь нет, считаем уже покрытым текущим API.

## Must Have

### 1. Живые `recentMatches` для страницы игрока

Сейчас frontend уже показывает на странице игрока блок последних матчей:

- группировка по турнирам;
- пары по раундам;
- счёт с точки зрения игрока;
- колода игрока и колода оппонента.

Но по текущему контракту backend возвращает `recentMatches: []`.

Нужно одно из двух:

1. Либо начать реально заполнять `recentMatches` в `GET /players/{id}`.
2. Либо сделать отдельную ручку `GET /players/{id}/matches`.

Минимально нужная структура для одного матча уже есть во frontend:

```json
{
  "tournament": {
    "id": 12,
    "title": "Legacy Дейлик 08.07.2026",
    "date": "2026-07-08",
    "format": { "id": "legacy", "name": "Legacy" }
  },
  "roundNumber": 1,
  "tableNumber": 3,
  "playerDeck": { "id": 7, "name": "Reanimator", "archetype": "Reanimator", "colors": ["B", "U"] },
  "opponent": { "id": 42, "name": "Иван Петров" },
  "opponentDeck": { "id": 9, "name": "Death and Taxes", "archetype": null, "colors": ["W"] },
  "playerScore": 2,
  "opponentScore": 1,
  "scoreText": "2-1",
  "result": "win"
}
```

Без этого страница игрока остаётся неполной относительно текущего интерфейса.

## Nice To Have

### 2. Более полные short-объекты колод в компактных блоках

Во многих местах frontend умеет показывать mana pips, а для этого нужны `colors`.
Сейчас часть вложенных объектов колод приходит только как `id + name`.

Желательно отдавать `colors` и, по возможности, `archetype` в следующих местах:

- `home.recentTournaments[].winner.deck`
- `home.topPlayers[].mostPlayedDeck`
- `home.popularMatchups[].deckA`
- `home.popularMatchups[].deckB`
- `players.results[].mostPlayedDeck`
- `tournaments.results[].winner.deck`

Это не блокирует показ таблиц, но влияет на точность текущего UI:

- цветные mana pips;
- единый вид ссылок на колоды;
- меньше fallback-состояний на фронте.

### 3. `appliedFilters` прямо из backend

Сейчас frontend сам достраивает названия выбранных фильтров через отдельные запросы
к справочникам, потому что backend фильтрует данные, но не возвращает нормализованный
человекочитаемый результат применённого набора фильтров.

Желательно отдавать в списках и summary-ответах:

```json
{
  "appliedFilters": {
    "city": { "id": "moscow", "name": "Москва" },
    "club": { "id": "edinorog_moscow", "name": "Единорог", "cityId": "moscow" },
    "format": { "id": "legacy", "name": "Legacy" },
    "tournamentType": "daily",
    "dateFrom": "2026-01-01",
    "dateTo": "2026-07-01"
  }
}
```

Где это особенно полезно:

- `GET /home`
- `GET /tournaments`
- `GET /players`
- `GET /players/{id}`
- `GET /decks`
- `GET /decks/{id}`

Это не обязательно, потому что frontend уже умеет жить без этого, но с backend-версией:

- меньше дополнительных запросов;
- меньше логики на фронте;
- чище заголовки, бейджи и описания выбранного периода.

## Not Needed Now

### 4. Новые summary-метрики

Для текущего UI не требуются новые агрегаты вроде:

- отдельного global game winrate вне страницы игрока;
- продвинутых ELO / рейтингов;
- распределений по городам;
- трендов по неделям;
- временных рядов для графиков.

Этого в текущем интерфейсе нет, поэтому backend не должен тратить на это время раньше
критичных пробелов.

### 5. Дополнительные публичные auth-ручки ради статистики

Для показа публичной статистики отдельные auth endpoints не нужны.

Auth нужен только для служебного раздела загрузки турниров, а не для:

- главной;
- турниров;
- игроков;
- колод;
- карточек сущностей.

### 6. Новые страницы статистики

Для текущего фронта не нужны отдельные ручки под:

- отдельную страницу matchup explorer;
- отдельную страницу metagame trends;
- сравнение двух игроков;
- сравнение двух колод.

Таких экранов в текущем UI нет.

## Короткий итог

Если backend хочет закрыть именно текущий статистический UI, приоритет такой:

1. Заполнить `recentMatches` для страницы игрока.
2. Обогатить короткие объекты колод `colors` и, по возможности, `archetype`.
3. По возможности начать возвращать `appliedFilters`.

После этого статистика на frontend будет максимально близка к тому виду, который уже
заложен в интерфейсе.

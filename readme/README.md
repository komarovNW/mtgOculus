# MTG Global Stats MVP Frontend

## Цель проекта

Веб-сайт для просмотра статистики турниров Magic: The Gathering. MVP ориентирован на публичный просмотр статистики без авторизации.

На первом этапе основной сценарий:

- пользователь открывает сайт;
- видит главную статистику по дефолтному срезу: `Москва`, `Legacy`, `все клубы`;
- может перейти на страницу турнира, игрока или колоды;
- админ/разработчик может вручную добавить турнир через форму импорта.

## Что должно быть готово в конце MVP

Сайт должен содержать:

- главную страницу статистики;
- список турниров;
- страницу конкретного турнира;
- список игроков;
- страницу конкретного игрока;
- список колод;
- страницу конкретной колоды;
- экран создания турнира для ручного MVP-импорта;
- mock-режим для разработки без backend;
- подключение к реальному API через env-переменные;
- loading, empty и error states для всех основных экранов;
- адаптивную верстку для desktop и mobile;
- production build и инструкцию по деплою.

## Рекомендуемый стек

Использовать:

- React;
- TypeScript;
- Vite;
- React Router;
- TanStack Query для API-запросов и кеширования;
- CSS Modules или Tailwind CSS — выбрать один подход и использовать последовательно;
- Recharts для диаграмм, если нужны графики;
- Vitest для unit-тестов.

Не усложнять MVP:

- не добавлять Redux без необходимости;
- не добавлять сложную дизайн-систему;
- не добавлять авторизацию;
- не добавлять SSR/Next.js, если в этом нет явной необходимости.

## Env-переменные

Создать файл `.env.example`.

Минимальные переменные:

```env
VITE_API_BASE_URL=http://localhost:8000/api/v1
VITE_USE_MOCKS=true
```

Локальная разработка без backend:

```env
VITE_USE_MOCKS=true
```

Разработка с backend:

```env
VITE_USE_MOCKS=false
VITE_API_BASE_URL=http://localhost:8000/api/v1
```

Production:

```env
VITE_USE_MOCKS=false
VITE_API_BASE_URL=https://<production-api-domain>/api/v1
```

## Mock-режим

Mock-режим обязателен. Фронт должен уметь полностью открываться без backend.

Правило:

- если `VITE_USE_MOCKS=true`, данные берутся из mock-файлов;
- если `VITE_USE_MOCKS=false`, данные берутся из реального API.

Примерная структура:

```text
src/shared/api/
  client.ts
  types.ts
  mocks/
    home.mock.ts
    tournaments.mock.ts
    players.mock.ts
    decks.mock.ts
```

## Основные страницы

```text
/                         Главная статистика
/tournaments              Список турниров
/tournaments/:id          Детальная страница турнира
/players                  Список игроков
/players/:id              Детальная страница игрока
/decks                    Список колод
/decks/:id                Детальная страница колоды
/admin/tournaments/create Создание турнира
```

## Общие требования к UI

- Интерфейс на русском языке.
- Стиль: чистый аналитический dashboard.
- Основной акцент: читаемые таблицы, карточки статистики, простая навигация.
- Все кликабельные сущности должны иметь `id`: tournamentId, playerId, deckId.
- Проценты показывать с одним знаком после запятой.
- Пустые значения показывать как `—`.
- Для малой выборки показывать бейдж `Малая выборка`.
- Таблицы на мобильных устройствах должны не ломать верстку: horizontal scroll или карточное представление.

## Порядок разработки

Агент должен идти по шагам:

1. Создать проект и базовую конфигурацию.
2. Настроить routing.
3. Создать env и mock-режим.
4. Создать API layer и TypeScript-типы.
5. Создать базовые UI-компоненты.
6. Сделать layout приложения.
7. Сделать главную страницу на mock-данных.
8. Сделать страницы турниров.
9. Сделать страницы игроков.
10. Сделать страницы колод.
11. Сделать экран создания турнира.
12. Подключить реальные API.
13. Проверить loading, empty, error states.
14. Проверить адаптивность.
15. Подготовить production build и инструкцию деплоя.

## Definition of Done для MVP

MVP считается готовым, если:

- сайт запускается локально;
- все основные страницы открываются;
- mock-режим работает;
- реальные API можно подключить через env;
- главная страница показывает summary, турниры, метагейм, performance колод, топ игроков и матчапы;
- detail pages открываются по id;
- форма создания турнира отправляет multipart/form-data;
- есть обработка загрузки, пустого состояния и ошибок;
- сайт собирается через `npm run build`;
- есть инструкция по деплою.
